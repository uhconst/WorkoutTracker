## Build & Test Commands

```bash
# Compile only (fast — no APK, catches errors quickly)
./gradlew :composeApp:compileDebugKotlin
./gradlew :composeApp:compileKotlinIosArm64   # if iOS code was touched
./gradlew :wearApp:compileDebugKotlin

# Full build
./gradlew build

# Unit tests — composeApp (no device needed)
./gradlew :composeApp:testDebugUnitTest
./gradlew :composeApp:testDebugUnitTest --tests "com.uhc.workouttracker.workout.presentation.ExerciseListViewModelTest"

# Unit tests — wearApp
./gradlew :wearApp:testDebugUnitTest

# Instrumented UI tests (requires connected device or emulator)
./gradlew :composeApp:connectedDebugAndroidTest

# Wear OS build
./gradlew :wearApp:assembleDebug
```

> **Known Gradle issue:** `testDebugUnitTest` fails with `Type T not present` due to an AGP/Gradle version incompatibility. Run tests from Android Studio instead, or use `compileDebugUnitTestKotlin` to verify test sources compile.

There is no CI/CD pipeline. Run tests locally before committing.

## Architecture

**Clean Architecture by feature**, targeting Android + iOS (`composeApp`) and Wear OS (`wearApp`).

Each feature (`authentication/`, `workout/`, `muscle/`) has three layers:
- `domain/` — Kotlin-only interfaces and models. No Android, no Supabase imports.
- `data/` — Implements domain interfaces. Contains DTOs, mappers, and Supabase calls.
- `presentation/` — ViewModel + Composable screen. ViewModel depends only on domain interfaces.

### Room (offline-first cache)

Room 2.7.x (KMP) is used on **both Android and iOS**. All Room code lives in `commonMain`:
- `core/db/entity/`, `core/db/dao/`, `core/db/relation/`, `core/db/mapper/` — shared schema
- `core/db/WorkoutTrackerDatabase.kt` — uses `@ConstructedBy(WorkoutTrackerDatabaseConstructor::class)` (KMP pattern; `actual` is generated by KSP for each target)
- Feature `data/local/Room*LocalDataSource.kt` — shared implementations

Platform modules only differ in how the DB is **constructed**:
- `androidMain/di/AndroidModules.kt` → `androidDatabaseModule` uses `Room.databaseBuilder` with Android `Context`
- `iosMain/di/IosModules.kt` → `iosDatabaseModule` uses `BundledSQLiteDriver` from `androidx.sqlite:sqlite-bundled`

KSP must run for all targets — `build.gradle.kts` has `kspAndroid`, `kspIosX64`, `kspIosArm64`, `kspIosSimulatorArm64`.

### Dependency injection

Koin. Three modules in `composeApp/.../di/Modules.kt`:
- `supabaseModule` — `SupabaseClient` singleton
- `dataModule` — all repository bindings, DAOs, `LocalDataSource` impls, `SyncManager`, `AuthRepository`, `CoroutineScope`
- `viewModelModule` — ViewModels registered with `viewModelOf(::MyViewModel)`

Platform entry points (`WorkoutTrackerApplication`, `MainViewController`) load `androidDatabaseModule` / `iosDatabaseModule` plus the three shared modules. Adding a new repository binding goes in `dataModule`; adding a new ViewModel goes in `viewModelModule`.

### Navigation

Type-safe Jetpack Navigation with `@Serializable` route objects in `NavRoute.kt`. The `ModalNavigationDrawer` in `App.kt` wraps the `NavHost`.

### Sync (offline-first)

`ExerciseSyncManager` (in `commonMain/core/sync/`) runs in `applicationScope`:
- **Realtime subscription** (muscle groups): `selectAsFlow` wrapped in `while (isActive) { runCatching { ... }.onFailure { delay(10_000) } }` — always retries on network failure, never crashes the app.
- **On-demand sync** (exercises): `syncExercises()` is called from repository write methods; callers must wrap in `runCatching`.

ViewModels observe Room Flows directly and never call network APIs. The UI shows cached data immediately on launch; sync errors are only surfaced if the cache is also empty.

## Key Patterns

**ViewModel state:** Always `MutableStateFlow` for state, `MutableSharedFlow` for one-shot events (snackbars, navigation). Never `mutableStateOf` in a ViewModel.

**Error handling:** Every `viewModelScope.launch` that calls a repository must use `runCatching { }`. Error messages shown in UI are always static strings — never `exception.message`. Only set `_error` when the Room cache is also empty; if cached data exists, sync failures should be silent.

**Realtime subscriptions** in `applicationScope` must use the retry pattern (see `ExerciseSyncManager`). An unhandled exception in a `SupervisorJob`-scoped coroutine crashes the app.

**Screens are stateless:** All state comes from parameters; all events are lambda callbacks. `FooScreen()` collects from the ViewModel and calls `FooLayout(...)` which is the composable used in previews and UI tests.

**Flow-derived state:** Use `combine()` in the ViewModel. Never derive state inside a composable from two separate `collectAsState()` calls.

## Wear OS

The watch app (`wearApp`) is read-only. The phone pushes `accessToken` + `refreshToken` to the watch via Wearable Data Layer at path `/supabase_session` (`WearSessionSync.kt` on the phone). On the watch:

- `WearSessionRepositoryImpl.importAndValidateSession()` reads the tokens, calls `supabase.auth.importSession(..., autoRefresh = false)`, and refreshes only if `expiresIn == 0L`. Use `autoRefresh = false` to avoid background refresh races.
- `WorkoutsViewModel` calls `sessionRepository.importAndValidateSession()` — it does **not** import `SupabaseClient` directly.
- Unit tests for `WorkoutsViewModel` live in `wearApp/src/test/` and follow the same `UnconfinedTestDispatcher` + MockK pattern as `composeApp`.

## Testing Conventions

Unit tests: `composeApp/src/androidUnitTest/` and `wearApp/src/test/`. Pattern across all ViewModel tests:

```kotlin
@Before fun setUp() {
    Dispatchers.setMain(UnconfinedTestDispatcher())
    // mock repos, construct VM
}
@After fun tearDown() { Dispatchers.resetMain() }
```

Mock repositories with MockK (`mockk<MyRepository>(relaxed = true)`). Use `coEvery` for suspend functions. Use Turbine (`vm.myFlow.test { ... }`) for asserting emissions over time. Never mock `SupabaseClient` directly.

For `observeX()` methods that return a `Flow`, mock with `MutableStateFlow` for full emission control:
```kotlin
private val myFlow = MutableStateFlow<List<Foo>>(emptyList())
every { repo.observeFoo() } returns myFlow
```

UI tests in `androidInstrumentedTest/` use the `FooLayout` composable (not `FooScreen`) so no ViewModel or DI is involved.
